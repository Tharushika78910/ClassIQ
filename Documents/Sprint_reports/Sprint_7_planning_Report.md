# Sprint 7 ( Testing Sprint )

---

## Sprint Goal

---

The goal of Sprint 7 is to test the ClassIQ system properly and make sure the application is functional, user friendly, and aligned with the project requirements. In this sprint, the team focuses on final functional and non-functional testing, bug fixing, quality checking with Jenkins and SonarQube, and updating the project documents based on the testing results.

## Selected Product Backlog Items

---

- Create the Sprint 7 test plan  
- Perform final unit testing for completed features
- Track bugs and issues found during testing
- Review and refine Trello user stories
- Run Jenkins build and Docker deployment
- Generate SonarQube report and improve code quality
- Conduct heuristic evaluation
- Conduct User Acceptance Testing (UAT)
- Document technical and architecture changes
- Update Trello, GitHub, and sprint report documentation

## Planned Tasks/Breakdown

---

- Prepare Test Plan

In Sprint 7, the team first prepares a test plan for the project. This includes the testing objective, resources, test environment, and the main testing tasks. The purpose of this plan is to guide the team during both functional and non-functional testing.

- Functional Testing and Bug Fixing

The team carries out final unit testing for the main features of the system to make sure the application works correctly after the code cleanup done in the previous sprint. During testing, bugs and issues are recorded in a tracking table, and the team fixes them one by one. The Trello user stories are also reviewed and updated according to the latest system functions.

- Build, Deployment, and Code Quality Checking

The project is then tested through Jenkins to confirm that the build is successful. Docker deployment is also checked to make sure the system can run correctly in the deployment environment. In addition, SonarQube is used to generate the static code analysis report, and the team works to keep the quality grades at A or at least B.

- Non-functional Testing and Documentation

For non-functional work, the team performs heuristic evaluation to check usability and User Acceptance Testing to confirm that the system meets user needs. If any technical or architectural changes are needed after testing, they are documented properly. Finally, Trello, GitHub, and the sprint report are updated with the completed work and changes made during the sprint.

## Team Capacity & Assumptions

---

The team members have knowledge of Java development, UI design, testing, database handling, Docker, Jenkins, and SonarQube. They are able to perform functional testing, identify and fix bugs, evaluate usability, run CI/CD related checks, and update project documentation based on the final test results.

## Definition of Done

___

A backlog item is considered done when the related feature or testing task is fully completed and verified. The application should run correctly without major errors, and the final unit testing should confirm that the implemented functions are working properly. All important bugs found during testing should be recorded and addressed. In addition, Jenkins build, Docker deployment, and SonarQube checks should be completed successfully. The heuristic evaluation, UAT results, and documentation updates should also be finished. Finally, the Sprint 7 report should be ready with the required evidence, screenshots, and team contribution details.
