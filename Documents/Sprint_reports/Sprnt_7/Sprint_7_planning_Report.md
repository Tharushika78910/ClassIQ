# Sprint 7 ( Testing Sprint )

---

## Sprint Goal

---

The goal of Sprint 7 is to test the ClassIQ system properly and make sure the application is functional, user friendly, and aligned with the project requirements. In this sprint, the team focuses on final functional and non-functional testing, bug fixing, quality checking with Jenkins and SonarQube, and updating the project documents based on the testing results.

## Selected Product Backlog Items

---

- Create a detailed test plan including objective, resources, test environment, and test tasks
- Perform final unit testing for completed features
- Track bugs and issues found during testing
- Review and refine Trello user stories
- Run Jenkins build and Docker deployment
- Generate SonarQube report and improve code quality
- Conduct heuristic evaluation
- Conduct User Acceptance Testing (UAT)
- Perform performance and security testing
- Document technical and architecture changes
- Update Trello, GitHub, and sprint report documentation

## Planned Tasks/Breakdown

---

- Prepare Test Plan

In Sprint 7, the team first prepares a detailed test plan for the ClassIQ system. This test plan is created to organize the testing process in a clear and proper way and to make sure all important parts of the system are checked. The plan includes the testing objective, resources, test environment, and the main test tasks that will be carried out during the sprint.

The objective of the test plan is to verify that the ClassIQ system works correctly, meets the project requirements, and is ready for final evaluation. It also helps the team check whether the system is stable, user friendly, reliable, and suitable for final submission.

The resources used for testing include the project source code, GitHub repository, Jenkins pipeline, Docker environment, SonarQube reports, test cases, bug tracking table, and team members who are involved in testing, evaluation, and documentation work.

The test environment includes IntelliJ IDEA, Java/JavaFX, MySQL database, Jenkins, Docker, SonarQube, GitHub, and Trello. This environment is used to run the project, test the system functions, verify the build and deployment, and review code quality.

The test tasks are divided into functional and non-functional testing. Functional testing includes final unit testing of the major system features, checking login and dashboard functions, validating localization features, testing database operations, and identifying and fixing bugs. Non-functional testing includes heuristic evaluation, User Acceptance Testing, performance testing, and security testing. Performance testing is used to check the responsiveness and speed of the system, while security testing is used to identify possible weaknesses such as input handling issues, access control problems, or unsafe data operations. This test plan helps the team clearly understand what will be tested, how it will be tested, and what tools and resources will be used during Sprint 7.

- Functional Testing and Bug Fixing

The team carries out final unit testing for the main features of the system to make sure the application works correctly after the code cleanup done in the previous sprint. During testing, bugs and issues are recorded in a tracking table, and the team fixes them one by one. The Trello user stories are also reviewed and updated according to the latest system functions.

- Build, Deployment, and Code Quality Checking

The project is then tested through Jenkins to confirm that the build is successful. Docker deployment is also checked to make sure the system can run correctly in the deployment environment. In addition, SonarQube is used to generate the static code analysis report, and the team works to keep the quality grades at A or at least B.

- Non-functional Testing and Documentation

For non-functional work, the team performs heuristic evaluation to check usability and User Acceptance Testing to confirm that the system meets user needs. Performance testing is carried out to observe system responsiveness, and security testing is used to identify possible vulnerabilities. If any technical or architectural changes are needed after testing, they are documented properly. Finally, Trello, GitHub, and the sprint report are updated with the completed work and changes made during the sprint.

## Definition of Done

___

A backlog item is considered done when the related feature or testing task is fully completed and verified. 
The application should run correctly without major errors, and the final unit testing should confirm that the implemented functions are working properly. 
The test plan should be completed with objective, resources, test environment, and test tasks. All important bugs found during testing should be recorded and addressed.
In addition, Jenkins build, Docker deployment, and SonarQube checks should be completed successfully. The heuristic evaluation, UAT results, performance and security testing, 
and documentation updates should also be finished. Finally, the Sprint 7 report should be ready with the required evidence, screenshots, and team contribution details.
